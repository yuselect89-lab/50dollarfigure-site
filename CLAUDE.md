# $50 FIGURE / YUSelect — Claude Code 運用ルール

このリポジトリは $50 FIGURE / YUSelect の公式サイト（静的HTML）です。
それとは別に、Yutaさんの在庫管理は Notion の「在庫トラッカー」データベースで行っており、
Yutaさんは在庫トラッカーをNotion上で直接編集せず、**Claude Codeとの会話で入力を完結させる**運用です。
以下のワークフロールールに従い、該当する発言を検知したら在庫トラッカーへの追加・更新を自動で行ってください。

## Notion在庫トラッカーの参照情報

- データソースURL: `collection://d7b55c79-19e2-45ba-bf41-81e72df196bf`
- プロパティ名には区分絵文字が付いています。**✍️ = 手入力項目 / 🔒 = 自動計算・自動取得項目**。
  ページの作成・更新時は必ず絵文字込みの正式なプロパティ名（例：`✍️ 商品名`、`🔒 ステータス`）を使うこと。

## パターンA：仕入れ報告（新規ページ作成）

Yutaさんが仕入れた商品について自然文で報告したら、在庫トラッカーに新規ページを1件作成する。

**発言例：**
> 「Uzaki-chanのスワイムスーツ、Taito製、1350円で仕入れた。状態は未開封、重さ500g」

**マッピングルール：**
- `✍️ 商品名` → 発言内の商品名
- `✍️ メーカー` → 発言内のメーカー名
- `✍️ 仕入れ値（円）` → 発言内の金額
- `✍️ コンディション` → 発言内の状態表現
- `✍️ 重量（g）` → 発言内の重さ
- `🔒 仕入れ日` → 発言があった当日の日付を自動設定
- `🔒 ステータス` → 「仕入れ済み」を自動設定
- `✍️ ブランド区分` → 発言内に明記がなければ「$50 FIGURE」をデフォルトとし、Yutaさんに確認する
- `✍️ シリーズ・作品名`、`✍️ 仕入れ先` など、発言に含まれない項目は空欄のままにし、後から埋められることをYutaさんに伝える

情報が不足している場合（金額や商品名が読み取れない等）は、ページを作らずYutaさんに聞き返すこと。

### eBayタイトル案・商品説明案の生成ルール

パターンAでページを作成する際、Yutaさんが出品用の文言も必要としている場合（または
出品準備の一環として求められた場合）は、以下のルールに従って「eBayタイトル案」と
「eBay商品説明案（HTML）」を生成する。ブランド区分（$50 FIGURE / YUSelect）に
関わらず、同じルール・同じテンプレートを使う。

**1. カテゴリ判定（最初に必ず行う）**
商品が以下のどちらに該当するか判定する：
- **カテゴリA：アニメ・キャラクターフィギュア**（クレーンゲーム景品、プライズ品など）
- **カテゴリB：トレーディングカードゲーム（TCG）の単品カード**

**2. Item Specifics 生成ルール**

*カテゴリA（フィギュア）の場合：*
- `Brand`：具体的なメーカー名を必ず記載（例：`BANDAI SPIRITS`、`FuRyu`、`TAITO`、`SEGA`）。空欄・汎用値は禁止
- `Type`：`Action Figure / Statue`
- `Character`：キャラクター名
- `Franchise`：作品名
- `Series`：商品ライン名（例：`Trio-Try-iT` / `BiCute Bunnies`）
- `Version`：衣装・バリエーション名
- `Material`：`ATBC-PVC, ABS`
- `Size`：おおよそのサイズ（cm）
- `Condition`：下記コンディションマッピング参照

*カテゴリB（TCGカード）の場合：*
- `Game`：ゲーム名（例：`One Piece TCG`）
- `Card Name`：`[キャラクター名]（[カードID] [レアリティ]）`
- `Character`：キャラクター名
- `Set`：正式な英語セット名
- `Language`：`Japanese`
- `Card Condition`：下記コンディションマッピング参照
- `Quantity`：`1 Card`
- `Graded`：`No`

**3. コンディションマッピングルール（Item Specifics用）**
- **TCGカード（$50 FIGURE：開封したパックから出たもの）：** 常に `Lightly Played / Played`
  にマッピングする。ただし、開封後すぐにスリーブ保管されたことが明記されている場合のみ
  `Near Mint or Better` とする
- **TCGカード（YUSelect：カードショップ等から状態表記付きで仕入れた単品）：** 仕入れ元
  （bigweb等）が示す状態表記をそのまま使う（例：美品／傷ありなど、仕入れ時にYutaさんが伝えた
  内容を反映する）。仕入れ元に状態表記が無ければ、$50 FIGURE同様 `Lightly Played / Played`
  にマッピングする
- **フィギュア（仕入れ品）：** 日本のリユースショップ経由で仕入れた場合、たとえ未開封・
  封印状態であっても `Used (Unopened / Like New Condition)` にマッピングする
  （消費者保護の取引区分に厳密に準拠するため）
- **YUSelect新品小売品（マンガ・雑誌単行本・CD等、書店・コンビニ等で新品購入したもの）：**
  `New` にマッピングする（リユース品ではないため、フィギュアと同じ「Used」扱いにしない）

**4. タイトル生成ルール**
- **構成：** `[作品名] [キャラクター名] [商品ライン/カードID] [レアリティ・バージョン] [特徴] Japanese`
- **文字数：** 80文字以内に収める
- **禁止語：** スパム的・規約違反ワードは使わない（例："Review"、"Sale"、"Cheap"、"Wow"、
  普遍的に当てはまらない場合の"Free Shipping"など）
- **表記：** Title Case、装飾記号なし、余分なスペースなし

**5. HTML商品説明テンプレート**
基本構造はカテゴリA・カテゴリBそれぞれについて、`docs/ebay_listing_prompt_template.md`
記載のHTML構造をそのまま使用する
（鑑定済みボックス、商品詳細リスト、箱の状態注記、配送案内、メッセージ歓迎ボックスなど）。

**ただし、テンプレート内の「EU & ITALY BUYERS」警告ボックスは、ブランド区分に関わらず
共通で以下の内容に統一すること：**

```html
<div style="background-color: #fdf2f2; border: 1px solid #f5c6cb; padding: 15px; margin-top: 20px; border-radius: 4px;">
  <p style="margin: 0; font-weight: bold; color: #721c24;">⚠️ IMPORTANT SHIPPING RESTRICTIONS NOTICE:</p>
  <p style="margin: 5px 0 0 0; color: #721c24; font-size: 14px;">
    We currently do not ship to EU countries, Alaska &amp; Hawaii, Russia, or most of South America
    (Argentina, Bolivia, Brazil, Colombia, Ecuador, the Falkland Islands, French Guiana, Guyana,
    Paraguay, Peru, Suriname, Uruguay, Venezuela). Chile is the one exception. If an order is placed
    from an excluded region, we will need to cancel the transaction.
  </p>
</div>
```

（警告ボックスの見出しは、対象地域が拡張されたことに合わせて「EU & ITALY BUYERS」から
「SHIPPING RESTRICTIONS NOTICE」に変更している。文面自体はYutaさん指定の通り）

**6. 実行ルール**
- 商品説明・特徴（傷、シール跡、レア度など）は、テンプレート内の該当ボックス
  （箱の状態注記など）に自然に統合する
- 出力は「Item Specifics」「eBayタイトル案」「HTML商品説明案」の3セクションに
  分けて、見やすく提示する
- 判断に迷うカテゴリ・コンディションがあれば、決め打ちせずYutaさんに確認する

## パターンB：出品報告（eBay URLからの自動取得・既存ページ更新）

Yutaさんが「これ出品した」等と共にeBayのURLを送ったら、該当ページを更新する。

**発言例：**
> 「これ出品した https://www.ebay.com/itm/xxxxx」

**注意（重要）：** eBayは`www.ebay.com`配下のページ（通常の商品ページ・Sold/Completed検索結果とも）を
WebFetch/WebSearchからのアクセスに対して403で弾く。**商品タイトル・価格はWeb取得できない前提で運用する。**
`🔒 eBay商品ID`はURL自体から抽出できる（例：`.../itm/127971786821` → `127971786821`）ので取得取得は不要。
価格は取得を試みて失敗したら、Yutaさんに一言で聞き返す（毎回聞き返すのが煩わしければ、発言時に価格も
一緒に書いてもらう運用でもよい）。

**処理手順：**
1. 在庫トラッカーの中から、商品名が近いページ、または直近で「仕入れ済み」ステータスのページを探し、どれを更新するかYutaさんに確認する（複数候補がある場合は必ず確認する。1件しかない場合は確認省略可）
2. URLから商品IDを抽出する。価格・タイトルはWeb取得を試み、失敗したらYutaさんに確認する
3. 以下を自動入力する：
   - `🔒 eBay商品ID`
   - `🔒 eBay URL`
   - `🔒 出品価格（USD）`
   - `🔒 ステータス` → 「出品中」に変更
   - `🔒 出品日` → 当日の日付

## パターンC：売却報告

Yutaさんが「売れた」と報告したら、該当ページを更新する。

**発言例：**
> 「Uzaki-chan売れた、$50、送料は4000円、レートは162円」

**マッピングルール：**
- `✍️ 実売上（USD）`、`✍️ 為替レート`、`✍️ 送料実費（円）` → 発言内の数値
- `✍️ 発送先国` → 発言に含まれていれば入力、なければ空欄
- `🔒 ステータス` → 「売却済み」に変更
- `🔒 売却日` → 当日の日付

情報が不足していれば、不足分だけYutaさんに聞き返すこと（全項目を毎回聞き返さない）。

## パターンD：ウェブサイトへの自動反映

在庫トラッカーで `🔒 ステータス` が「出品中」のページは、`sync_inventory.py` と
GitHub Actions（`.github/workflows/sync-inventory.yml`）によって自動的に
`index.html` の「Today's Shelf」に反映される（1日1回の定期実行、または手動トリガー）。

**Yutaさんが行うこと：**
- 出品報告（パターンB）の際、在庫トラッカーの `✍️ 商品写真` プロパティに
  **1枚目＝AI生成プロモ画像（SNSと同じもの）、2枚目＝実物の箱写真（背景透過PNG、任意）**
  の順でアップロードする。サイトの商品カードに表示されるのは常に1枚目（AI画像）
  ——実物写真・動画はeBayに飛んで初めて見える、という運用は変わらない
- 2枚目（実物箱写真）をアップロードしておくと、商品カードの写真をクリックした際の
  拡大表示（ライトボックス）が2枚のスライドになり、2枚目でその実物箱写真が見られる
  ようになる。透過PNGを自動でサイトの紙色背景（`#F7F3EC`）に合成し、影を付けて
  馴染ませる処理をしているので、Yutaさん側は背景加工済みの透過PNGを用意するだけでよい
- 各スライドには「AI-generated promo image」「Actual item you receive (unopened)」
  という注記が自動で付き、世界観画像と実物を混同されないようにしている
- 写真は自動的に取得・圧縮（リサイズ＋JPEG化）され、`assets/products/` に実ファイルとして
  保存・`index.html` から参照される（以前はBase64埋め込みだったが、ファイルサイズと
  表示速度の都合で実ファイル方式に変更済み）。不要になった古い写真ファイルは同期のたびに
  自動削除される
- 1枚目が未アップロードの場合は、プレースホルダー画像（`assets/placeholder.jpg`）が
  暫定的に表示される。2枚目（箱写真）は無くてもよく、その場合ライトボックスは1枚表示のまま

**反映される項目：**
- `✍️ 商品名`、`✍️ メーカー`、`✍️ シリーズ・作品名`、`✍️ コンディション`
- `🔒 eBay URL`
- `✍️ 商品写真`の1枚目（未設定時はプレースホルダー）、2枚目があればライトボックスの
  2枚目スライドとして反映

**注意：** ステータスが「出品中」以外（仕入れ済み・売却済み・キャンセル等）のページは
ウェブサイトに表示されない。売却済みになったら自動的にサイトから消える。

### Netlifyクレジットの節約について

サイトはNetlify（有料プラン、月1000クレジット）でホストされている。Netlifyは
**本番デプロイ1回につき15クレジット消費**するため、無駄なデプロイを避けることが重要。

- `netlify.toml` に `ignore` 設定があり、**`index.html`・`assets/`・`thanks.html` が
  実際に変わったコミットのみデプロイが走る**ようにしてある。`CLAUDE.md`・
  `sync_inventory.py`・`.github/workflows/` などサイトの見た目に影響しない変更は
  自動的にデプロイをスキップする
- `sync_inventory.py` 自体も、在庫トラッカーに実際の変化（新規出品・売却・写真変更等）が
  無い日はコミット・pushをしないため、無駄なデプロイは発生しない
- Claude Codeが作業する際も、関連する変更はなるべく1つのコミットにまとめてpushし、
  小さな修正のたびに何度もpush・手動トリガーすることは避けること

## Outlet Corner（アウトレット商品のバンドル販売）

開封動画で使用した商品や、コンディションが良くない商品を、単品では売らず、
「出品中」の$50商品との抱き合わせ（+$25）で販売する仕組み。

**運用フロー：**
1. 在庫トラッカーで対象商品の `✍️ アウトレット対象` にチェックを入れる
   （`✍️ アウトレット価格（USD）` は未設定ならデフォルト$25）
2. `sync_inventory.py` が自動的にウェブサイトの「🏷️ Outlet Corner」セクションに反映する
   （商品写真・コンディションも通常商品と同じ仕組みで自動取得・翻訳される）
3. Outlet Corner内の各商品カードには、現在「出品中」の$50商品をプルダウンで選べる
   フォームが付いており、そのプルダウンの中身は同期のたびに最新の出品中商品リストに
   自動更新される（売り切れた商品が古いまま残ることはない）
4. お客さんがプルダウンで抱き合わせたい商品を選び、名前/ハンドルネームと連絡先
   （eBayユーザー名またはメール、任意項目）を入力して送信すると、
   Formspree（`sync_inventory.py` 内の `FORMSPREE_ENDPOINT`）経由でYutaさんのメールに通知が届く
5. Yutaさんが該当eBayリスティングを手動で編集（+$25、写真追加、タイトルを
   「〜様専用セット」に変更）し、連絡先が入力されていればeBayメッセージ／メールで
   お客さんに知らせる。連絡先が空欄だった場合は、お客さん自身が24時間以内に該当リスティングの
   タイトルを確認して購入する運用（フォームの注記に明記済み）
6. 予約は24時間以内が期限。連絡先ありなら連絡してその期限内に、連絡先なしならお客さんの
   自己確認で期限内に。過ぎたら他のお客さんに開放する旨をフォーム上に明記

**注意：** `FORMSPREE_ENDPOINT` はプレースホルダーのままだと通知が届かない。
Yutaさんが Formspree でフォームを作成し、発行されたエンドポイントURLに置き換える必要がある。

**Yutaさんへの通知を見逃さないために：** Formspreeの無料プランはWebhook/Discord/Slack連携が
使えず（Personalプラン以上、月$10〜で解禁）、通知はメールのみ。スマホですぐ気づきたい場合は、
①スマホのメールアプリでFormspreeからのメールにフィルタ・強制通知設定をする（無料）、
②Formspreeを有料プランに上げてDiscordのプライベートチャンネルに転送する（月$10〜、
通知が最も目立つ）のどちらかがおすすめ。

## パターンE：SNS投稿キャプション自動生成

毎週日曜日、翌週分（月〜金）のコンテンツカレンダーのエントリに対して、
キャプション文案とハッシュタグを自動生成し、ステータスを「アイデア」→「下書き」に
更新する。Yutaさんは生成AIの動画・画像そのものは自分で作成し、キャプション・
ハッシュタグ・テキスト投稿（知識系・透明性ネタ）の文章をClaude側が担当する。
Yutaさんが下書きを確認・修正した上で、Buffer等で予約投稿する運用。

**共通ルール：**
- 文字数はX（無課金アカウント）を基準に、**180字程度**に収める
- トーン・文体は `index.html` の「About」文と統一（日本在住・実際に目利きしている・
  気取らない一人称の語り口）
- 過去数週間分の `キャプション（英語）` を確認し、同じネタ・同じ言い回しの繰り返しを避ける
- `キャプション（英語）` を生成したら、Yutaさんが内容を確認しやすいように
  `キャプション（日本語訳）` にも同じ内容の日本語訳を必ず書き込む（投稿はあくまで
  英語のみ。日本語訳は確認用でSNSには投稿しない）

**①AI生成プロモ・②開封動画のキャプション：**
- 対応する商品は、コンテンツカレンダーの `Related to 在庫トラッカー (関連コンテンツ)`
  プロパティでリンクされた在庫トラッカーのページを参照して判断する（商品名・メーカー・
  シリーズ・コンディションはここから正確に拾う）
- `使用素材` プロパティは、商品名などの事実情報ではなく、**その投稿ならではの補足メモ**
  （例：「背景は公園」「桜並木バージョン」「開封動画」など、演出意図やコンテンツ型の
  目印）を書く場所。空欄でも `コンテンツ型` と商品リンクがあれば最低限のキャプションは
  生成できるが、雰囲気が伝わる一言があった方が質は上がる
- `Related to 在庫トラッカー` が空欄のエントリは生成をスキップし、実行後の報告で
  Yutaさんに商品をリンクするよう伝える
- **①AI生成プロモ**は「AI生成の世界観イメージ＋外箱写真」の2枚セットが基本構成
  （仕入れが未開封中心のため、中身のイメージ喚起はAI背景・フィギュア本体は実物準拠で行う）。
  キャプションには、世界観イメージと実物（未開封で届く商品）が違うものだと一目で伝わる
  一言を必ず入れる（期待値のズレを防ぐため）。外箱写真は傷やシール跡を隠すような
  加工・選び方をしない
- **②開封動画**は仕入れが未開封中心の都合上、**毎週安定して出せるコンテンツではない**。
  月水金の枠を機械的に①②交互で埋めようとせず、Yutaさんが実際に開封した週だけ
  `使用素材` に記載される想定で扱う（無理に②を毎週埋めない）。開封した商品は
  その後Outlet Cornerで「$50商品との抱き合わせ（+$25）」としてのみ販売されるため、
  該当週のキャプションでは「開封後はOutlet Cornerで$50商品とセット購入できる」旨を
  自然に触れてよい

**④知識系のキャプション：**
- フィギュアの種類・メーカー史・作品トリビアなど、事実に基づく内容を扱う
- **特定の作品名・メーカー・固有の史実に触れる場合は、必ずWeb検索で事実確認してから
  書く**（生成AIの知識だけに頼って断定的に書かない）
- 一般的な collecting culture の知識（コンディションの見方、フィギュアの種類の違いなど）
  で固有名詞を伴わない場合は検索なしでも可

**⑤透明性ネタのキャプション：**
- **架空のエピソードは絶対に作らない。** 必ず「透明性ネタ帳」データベースの実データのみを
  元に書く
- 生成手順：
  1. 透明性ネタ帳（`collection://b356e77a-9f12-4bda-946d-7d2154890963`）から、
     `ステータス` が「回答済み・未使用」のページを古い順に1件探す
  2. 見つかった場合：その `質問`・`回答` の内容を元に投稿文を作成し、使用後は
     `ステータス` を「使用済み」に変更、`使用日` に当日の日付、
     `使用した投稿` に生成したコンテンツカレンダーのページを紐付ける
  3. 見つからない場合：無理に生成せず、該当エントリのステータスは「アイデア」のまま
     残し、実行後の報告でYutaさんに知らせる
  4. 使用の有無にかかわらず、**新しい具体的な質問を1件、透明性ネタ帳に追加**する
     （ステータス「未回答」）。抽象的な質問（「何か面白いことあった？」等）ではなく、
     具体的なシーン・数字・やりとりを尋ねる質問にすること
     （例：「今週の仕入れで一番印象に残ったやりとりは？」など）

**実行後の報告：**
- 生成した投稿の一覧、スキップした投稿とその理由（商品未リンク・ネタ帳に回答なし等）、
  透明性ネタ帳に追加した新しい質問、を簡潔にまとめる

## パターンF：YUSelect 無在庫・有在庫ワークフロー

YUSelectは「無在庫（真の無在庫、売れてから仕入れる）」と「有在庫（先に少量仕入れる）」の
2本立てで運用する。判断基準はジャンルではなく**仕入れ値の高さ**：

- **無在庫**：仕入れ値が高い商品、または回転が遅い商品。低単価・高回転（週刊誌・単行本など）も、
  逆に高単価で売れるまで遅いものも、どちらも対象になり得る。共通点は「在庫を長期間抱える
  資金リスクを避けたい」こと
- **有在庫**：仕入れ値が安ければ（目安：数百円程度）、多少売れるのが遅くても在庫を持ってよい。
  資金拘束・ロスが小さいため。例：単価の低いTCG単品カード、100円ショップ雑貨

**候補リサーチ：**
「YUSelect 無在庫候補」データベース（collection://a3779570-a0a9-4f3f-b151-86f0ef53049e）で
候補を管理する。毎週日曜21時JSTのRoutineが、再販売価格維持制度対象（書籍・雑誌・CD等）や
チェーン均一価格の商品を中心に3〜5件、eBayのSold検索リンク付きで提案する（ステータス=提案中）。
Yutaさんが内容を確認し、ステータスを「承認」または「却下」に変更する。

**在庫トラッカーとの連携：**
候補が承認され、Yutaさんが実際に仕入れたら、通常の**パターンA（仕入れ報告）**で
在庫トラッカーに新規ページを作る（`✍️ ブランド区分` = `YUSelect`）。「YUSelect 無在庫候補」の
該当ページの`在庫トラッカー連携`で紐付け、ステータスを「出品済み」に更新する。以降は
パターンB（出品報告）・パターンC（売却報告）をそのまま使う。

**無在庫の場合の注意：** 売れてから仕入れに行く運用のため、Yutaさんが在庫トラッカーに
仕入れ報告する前にeBayへ出品するケースがある（在庫が無い状態で先に出品ページを作るため）。
その場合もパターンAで仕入れ報告を受けたら通常通りページを作成し、既にeBay URLがあれば
パターンBの内容も同時に反映してよい。

**このワークフローの限界（正直な制約）：**
- eBayの商品ページ・Sold検索結果は自分のWebFetch/WebSearchからは403で弾かれ、直接読めない。
  需要シグナルはURLを貼るところまでが自分の役割で、実際の数字はYutaさんの確認が必要
- bigweb等のカードショップサイトも同様に直接読めない。仕入れ値・在庫状況はYutaさんから
  教えてもらう前提で運用する
- 「売れてから仕入れに行く」実作業（購入・梱包・発送）は完全に人間の作業であり、自動化できない
- 将来的にeBay Developer Programでアプリ登録し、Browse API（アクティブ出品の検索・詳細取得、
  特別審査不要）のキーを発行してもらえれば、この制約の一部は改善できる見込み
  （ただしSold/Completed検索はMarketplace Insights APIという別の審査制APIが必要で、
  これは今のところ通っていない）

## 共通ルール

- 曖昧な情報（金額の桁、商品名の表記揺れなど）は、推測で埋めずに必ずYutaさんに確認する
- 1回の発言で複数商品に該当しうる場合は、対象を確定させてから処理する
- 処理が完了したら、「何を」「どのページに」反映したか、簡潔に報告する
